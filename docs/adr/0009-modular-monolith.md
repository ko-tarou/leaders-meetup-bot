# ADR-0009: モジュラーモノリスの採用（マイクロサービスは不採用）

- Status: Accepted
- Date: 2026-07-14

## Context

DevHub Ops は 1 Worker + 1 D1 のモノリスとして成長してきた（アクション 17 種、services 60 ファイル超）。
カンファレンス 2027 のガント/タスク管理（`gantt_tracker`）を追加するにあたり、
「機能ごとにサービスを分割すべきか」を検討した。

検討した選択肢:

1. 本物のマイクロサービス化（Worker 分割・DB 分割）
2. モジュラーモノリス（同一 Worker 内でモジュール境界を規約として導入）
3. 現状のフラット構造のまま追加

## Decision

**モジュラーモノリス（選択肢 2）を採用する。** 新機能は `src/modules/<機能>/` に閉じたモジュールとして実装し、`gantt_tracker` をその第 1 号とする。既存コードの全面リファクタは行わない（新規部分からモジュール構造を適用する）。

### モジュール境界ルール（規約）

- 配置: 1 機能 = `src/modules/<name>/`（例: `src/modules/gantt/`）。ルーター・サービス・型をモジュール内に閉じる。
- 依存方向: モジュール -> core（`src/db` / `src/types` / `src/middleware` / `src/services` の共有基盤）は可。**core -> モジュールの import は、`src/routes/api.ts`（ルーター登録の単一合流点）のみ許可。**
- モジュール間: モジュール同士の直接 import は禁止。共有したくなったら core に降ろす。
- migration: モジュール由来の migration はファイル名に名前空間を含める（例: `0077_gantt_tracker.sql`）。番号は全体で単調増加のまま（D1 の適用順を壊さない）。
- テスト: モジュールのユニットテストは `test/modules/<name>/` に置く。

## Consequences

- 良い点: デプロイ・運用・監視は 1 Worker のまま（無料枠・非力 PC 運用に適合）。境界が規約として明文化され、将来の切り出し（本当に必要になったら）が容易。
- 悪い点: 境界はコンパイラで強制されない（レビューで守る）。既存コードはフラットなまま当面残る。
- 不採用理由（マイクロサービス）: 運用者 1 人・トラフィック極小の内部ツールに対して、Worker/DB 分割は認証・デプロイ・整合性のコストが便益を大きく上回る。
