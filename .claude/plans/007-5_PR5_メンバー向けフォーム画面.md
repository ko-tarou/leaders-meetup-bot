# PR5: メンバー向けフォーム画面

## 目的

メンバーが自分専用 magic-link（`/whitelist/:token`）を開いて、
誘いたい人のフルネームを add/remove・提出できる公開フォーム画面を実装する。
**自分のリストだけ**が見え、プライバシー注記を明示する。

## 前提

- PR1（型）+ PR2（公開 API `GET/POST /api/whitelist/:token`）完了済み。
- 公開ページ判定は `frontend/src/App.tsx`（~48-80 行、`/apply` `/interviewer-form` `/participation` `/public` を
  公開ページとして扱う分岐）に `/whitelist` を追加する。
- ルーティングは React Router の `<Route>`（既存 `path="/participation/:eventId"` 等と同じ階層）。

## 変更内容

### 1. `frontend/src/pages/WhitelistFormPage.tsx`（新規 ~140 行）

```tsx
// route: /whitelist/:token
// 1. マウント時 GET /api/whitelist/:token → { displayName, names, submitted }
//    - 404 → 「リンクが無効です」エラー画面。
// 2. names をチップ表示（× で削除）。テキスト入力 + Enter / 「追加」ボタンで追加。
//    - クライアント側で trim、空・正規化後重複を弾く（サーバ側でも再検証）。
// 3. 「提出する」ボタン → POST /api/whitelist/:token { names }。
//    - 成功 → 「提出しました」トースト / submitted バッジ更新。
// 4. プライバシー注記を常時表示（下記コピー）。
```

UI 構成:
- ヘッダー: 「🤝 ホワイトリスト登録」+ `{displayName} さん` の表示。
- プライバシーカード（最上部、目立つ枠）:
  > 🔒 このリストは**あなただけ**が閲覧・編集できます。
  > 全員が同じ名前を挙げたときだけ、**誰の名前かは伏せて**「誘いましょう」と通知されます。
  > このリンクは他の人と共有しないでください。
- 入力エリア: チップリスト + テキスト入力 + 追加ボタン。
- フッター: 「提出する」ボタン（提出済みでも再提出で上書き可、その旨の補足）。
- 空状態: 「まだ名前が登録されていません。誘いたい人のフルネームを追加してください。」

### 2. ルーティング登録（`frontend/src/App.tsx`）

- 公開ページ判定に `pathname.startsWith("/whitelist")` を追加。
- 公開 `<Routes>` ブロックに:
  ```tsx
  <Route path="/whitelist/:token" element={<WhitelistFormPage />} />
  ```

### 3. API クライアント

既存の fetch ラッパ（`frontend/src/lib/api` 等）に合わせて GET/POST を呼ぶ。
**admin token は付けない**（公開エンドポイント）。

## UX 要件（デザイナー #7 への対応）

- プライバシー注記を最初に読ませる（安心感）。
- チップ UI で add/remove を直感的に。Enter 追加・× 削除。
- 重複・空をクライアントで弾き、フィードバックを出す。
- 提出後の状態が分かる（「提出済み」バッジ + トースト）。
- 名前は本人入力のフルネームをそのまま表示（ID は登場しない）。
- モバイル幅で崩れない（既存公開フォームのレスポンシブに準拠）。

## テスト

`frontend/src/pages/__tests__/WhitelistFormPage.test.tsx`（vitest + testing-library、可能な範囲）:

## 設計判断メモ

- **クライアント正規化はベストエフォート**。真実はサーバ（PR2 の `normalizeName`）。
  クライアントは UX 改善（即時重複検知）のため軽く trim/重複チェックする程度。
- **再提出 = 上書き**。サーバが delete-then-insert で置換するので、画面側は「現在のリスト全体」を送る。
- 提出済みでも編集・再提出を許す（メンバーが後から人を足せる）。再提出のたびにサーバ側 consensus が再計算される。
