# 006 大規模リファクタ 全体計画（DevHub Ops）

## 背景

DevHub Ops は POC から運用プラットフォームへ拡張する過程で急成長した結果、以下の負債を抱えている。

| フロントエンド LOC | 約 26,697 行 / `frontend/src/**` | `wc -l` 実測 |

> 注: 旧見積もり「interactions.ts 約48,000行」は本計画作成時の実測（1,350 行）と大きく乖離していたため、本計画では実測値を採用する。BE 全体が約 18,884 行なので 48k 行のファイルは存在し得ない。

テストが 0 件の状態で大規模な構造変更を行うと、本番（Cloudflare Workers / D1）で回帰事故を起こすリスクが極めて高い。本計画は **安全網（テスト基盤＋現状の振る舞いを固定する characterization テスト）を最優先で構築してから**、段階的に 4 層アーキテクチャへ移行する。

## 目的

- ドメイン／ビジネスロジックを純粋関数として抽出し、テストで保護する
- 実用的 4 層アーキテクチャ（domain / application / infrastructure / presentation）へ分離する
- 外部依存（D1 / Slack / Gmail / GCal / Gemini）をインターフェース越しの adapter / Repository に隔離する
- 重複コード（Slack block builder / 通知 builder / メールテンプレ render）を統合する
- 巨大ファイルを振る舞い不変のまま分割する
- **回帰ゼロ**で完了する

## 絶対制約（厳守事項）

1. **既存機能を絶対に壊さない** — すべての PR は振る舞い保存（behavior-preserving）。回帰テストで証明できないものは出さない
2. **既存データを絶対に消さない** — migration は前方のみ。`DROP` / `ALTER ... DROP` / 破壊的 `UPDATE` 禁止。Phase 0〜4 では原則 migration を追加しない（必要時はユーザー承認必須）
3. **使用技術を極力変えない** — Vitest 等のテストツール追加は可。フレームワーク刷新（Hono / Drizzle / React / Vite から別物へ）は禁止
4. **各 PR ≤ 200 行** — 超えそうなら分割。1 PR = 1 エージェント
5. **各 PR でビルドが確実に通る** — `npm run typecheck` + `npm run build:frontend` を必須ゲート
6. **各フェーズ境界で 7 人格マルチレビューを過剰なほど実施し結果をユーザーに報告** — 承認を得てから次フェーズへ
7. **中間状態を本番に出さない** — フェーズが完成し回帰テスト全 green かつマルチレビュー承認まで本番デプロイしない（後述「移行中の本番デプロイ方針」）

## アーキテクチャ目標（実用的 4 層）

```
presentation : Hono routes (src/routes/**) / React (frontend/src/**)
       │  ← usecase を呼ぶだけ。ビジネス判断を持たない
application  : usecase（オーケストレーション）
       │  ← domain を組み合わせ、Repository/Adapter を介して I/O
domain       : 純粋関数・ビジネスルール（I/O なし、テスト容易）
infrastructure: D1=Repository, Slack/Gmail/GCal/Gemini=Adapter（インターフェース実装）
```

- 過剰な DDD（全域で集約／値オブジェクト）は採らない。値オブジェクト・不変条件は **必要箇所のみ**（例: ロール親子 invariant、日程候補生成ルール）
- 既存に存在する純粋寄り関数（`generateCandidateDates` / `computeSyncDiff` / `shouldStartPoll` / `readEmailTemplates` / `computeTargetRoleIds` 等）は domain 層の足がかりとして優先的に保護・移設する

## 実行戦略

- **並行エージェント数 = 2**。`isolation: "worktree"` 必須（共有 WD での `git checkout` 衝突防止）
- **PR 順序と依存グラフ**:
  - Phase 0: 0-1 完了が 0-2〜0-7 の前提（0-1 後は 0-2〜0-7 を 2 並行）
  - Phase 1〜3: 前フェーズの回帰テスト全 green かつマルチレビュー承認が次フェーズのゲート
  - Phase 4 は Phase 0 完了後なら Phase 1〜3 と独立に進行可（FE/BE 非干渉のため）。ただし並行上限 2 を守る
- **フェーズ間ゲート**: 「前フェーズ回帰テスト全 green」かつ「7 人格マルチレビュー承認（ユーザー報告済み）」を満たして初めて次フェーズ着手

依存グラフ（概略）:

```
0-1 → {0-2..0-7} → Phase1(1-1..1-6) → Phase2(2-1..2-5) → Phase3(3-1..3-3)
                 → Phase4(4-1,4-2) → {4-3..4-7}   （Phase4 は 0-1 後に独立着手可）
```

PR 総数の概算: Phase 0 = 7、Phase 1 = 6、Phase 2 = 5、Phase 3 = 3、Phase 4 = 7 → **計 28 PR 前後**（粒度調整で増減あり）。

## リスクと緩和策

| # | リスク | 緩和策 |
|---|---|---|
| 1 | テスト 0 からの開始 — 安全網が無いまま構造変更すると本番事故 | Phase 0 を最優先・本番コード非変更で安全網を先に構築。Phase 1 以降は Phase 0 テストを gating |

## スコープ外 / 非目標

- 機能追加はしない（振る舞い保存が大原則）
- 技術スタック刷新はしない（Hono / Drizzle / React / Vite / Workers / D1 を維持。テストツール追加のみ可）
- 一律 100% カバレッジは追わない（脆い・反パターン。ドメインは実質 100%、ルートは統合、UI はスモーク）
- 仕様改善・歪な仕様の修正はしない（characterization で固定し別 backlog へ）
- DB スキーマ変更・migration 追加はしない（原則。必要時はユーザー承認）
- 005 ハードニングスプリントで扱う品質改善（認証・冪等性等）とは別軸。重複する論点は 005 完了状態を前提に進める
