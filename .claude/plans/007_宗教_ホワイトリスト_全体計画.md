# 007 宗教（ホワイトリスト）— 全体計画

## ビジョン

新しいイベント「宗教」(type=`project`) を立ち上げる。このイベントは
**別の Slack ワークスペース**に紐づく（プラットフォームは既に複数ワークスペース対応済み：
`workspaces` テーブルがワークスペース毎に暗号化済み bot_token + signing_secret を保持し、
`POST /workspaces` で登録する）。

イベントは 2 つのアクションを持つ:

| アクション | action_type | 状態 |
|---|---|---|
| メンバー | `role_management` | **既存をそのまま再利用**（新規実装なし） |
| ホワイトリスト | `whitelist`（**新規**, label「ホワイトリスト」🤝） | 本施策で実装 |

### ホワイトリスト機能の概要

各メンバーが Web フォームから、**一緒に開発したい / 誘いたい人のフルネーム**（自由記述・複数可）を
**自分だけが見える**形でプライベートに登録する。
**全メンバーが同じ名前を挙げたとき（厳密な全会一致）**、システムが設定済みチャンネルに
「その人を誘いましょう」という Slack 通知を投稿する。
個人のリスト内容は本人以外（admin / 開発者含む）誰も閲覧できない。

## プライバシーモデル =「実用的な暗号化」（ユーザー承諾済み）

ユーザーは真のゼロ知識ではなく「実用的な暗号化」を明示的に選択した。**理由を正直に記録する**:

- 登録は Web フォームであり、入力値は平文で Worker に届く。
- 全会一致検出には全メンバーの選択を読む必要がある。
- → 「Web フォーム + 合意形成」を保ったまま **admin が一切覗けない（ゼロ知識）設計は原理的に不可能**。

代わりに以下を採用する:

1. リスト内容は保存時に **AES-256-GCM で暗号化**する（既存 `src/services/crypto.ts` の
   `encryptToken` / `decryptToken` と既存 wrangler secret `WORKSPACE_TOKEN_KEY` を再利用）。
   暗号文形式は既存と同一の `"iv_b64:ct_b64:tag_b64"`。
2. **いかなる admin UI でも個人のリストを表示しない**（件数すら出さない）。
3. 復号は**全会一致チェックの瞬間だけメモリ上で transient に**行う。

### 限界を正直に明記する（ドキュメントに残す）

- (1) Worker と鍵を握る**開発者は技術的には復号可能**。
- (2) magic-link は **bearer credential**。リンクを知る者は誰でもそのメンバーのリストを閲覧・編集できる
  （既存の参加届フォームと同じ posture）。**リンク再発行（rotate）で緩和**する。
- (3) **小 N 推論**: メンバーが少数だと、全会一致結果から他者の選択を部分的に推測できてしまう。

## 認証 = メンバー毎 magic-link（参加届フォームのパターン再利用）

- メンバー毎に **32 バイトのランダム hex token**（URL に含める）。
- メンバーは自分専用 URL を開く → Web フォームは**自分のリストだけ**を表示。
- 新しいログインシステムは作らない。プラットフォームには元々ユーザー毎認証は無く、
  admin ルートは共有の `x-admin-token` のみ。

> 既存の token 生成例: `src/routes/api/interviewers.ts` の `generateFormToken()`
> （24 バイト hex）。本施策は **32 バイト** に拡張する。

## 全会一致セマンティクス（厳密 — ユーザー確認済み）

- **メンバー集合** = ホワイトリストアクションに設定した role の現在メンバー
  （設定 `roleId` の `slack_role_members` を読む）。
- ある名前が「全会一致」となるのは、**すべての現 role メンバーが**
  `whitelist_members` 行を持ち `submitted_at` がセット済みで、**かつ**
  **その名前が全員の復号済みリストに含まれる**場合のみ。
- role メンバーが 1 人でも未提出（または whitelist 行なし）なら、**全会一致通知は一切発火しない**（待つ）。
- **未提出者への催促/リマインド通知は無し**（ユーザー却下）。未提出状態は **admin UI でのみ**表示。

## 名前マッチング

各名前を **Unicode NFKC 正規化 + trim + 内部空白の連続を 1 つに圧縮**してから完全一致比較する。
**既知の限界**: タイプミス・表記揺れはマッチしない（ドキュメントに明記）。

## 全会一致トリガ

- **フォーム提出（POST）成功のたびに再計算**する。
- 通知の重複は `whitelist_unanimous` テーブル（`UNIQUE(event_action_id, name_normalized)`）で抑止し、
  同じ名前を二度通知しない。
- **Slack 失敗は fail-soft**（ログのみ。保存はロールバックしない）。

## リンク配布

- admin UI から各メンバーのリンクを **「リンクをコピー」ボタン**で取得。
- **「リンク再発行（rotate）」ボタン**でローテーション。
- bot からメンバーへの **DM は送らない**（ユーザー却下）。

## admin UI 原則（プロジェクト規約）

- **ID は絶対表示しない。名前で表示する。**
- ホワイトリストタブが表示するもの: メンバー display_name / 提出済み・未提出 status /
  copy-link・rotate-link ボタン / 既に通知済みの全会一致結果一覧。
- **個人のリスト内容・件数は一切表示しない。**

## データモデル（新規 3 テーブル / 連番 ~0059〜0061）

> PR1 で最新マイグレーション番号を確認すること。**現在の最新は 0058** のため新規は 0059〜0061 を想定。

| テーブル | 用途 |
|---|---|
| `whitelist_members` | id PK / event_action_id FK→event_actions ON DELETE CASCADE / slack_user_id / display_name / token UNIQUE / submitted_at nullable / created_at / updated_at。UNIQUE(event_action_id, slack_user_id)、token に index |
| `whitelist_entries` | id PK / member_id FK→whitelist_members ON DELETE CASCADE / name_encrypted / created_at。member_id に index。**名前 1 つ = 1 行**、AES-256-GCM 暗号文 `"iv:ct:tag"` |
| `whitelist_unanimous` | id PK / event_action_id FK→event_actions ON DELETE CASCADE / name_normalized / notified_at。UNIQUE(event_action_id, name_normalized)。名前は**既に公開ブロードキャスト済みなので正規化平文で保存**（dedup UNIQUE 制約を可能にする） |

## API サーフェス

### 公開（admin-auth 除外。`src/routes/api.ts` の `/participation/*` `/apply/*` と同様に登録）

- `GET /api/whitelist/:token`
  → そのメンバー自身の**復号済み名前リスト** + 最小限のアクション/メンバー情報（display_name）を返す。token を照合。
- `POST /api/whitelist/:token`
  → body `{ names: string[] }` → バリデーション → 暗号化して entries を置換 → `submitted_at` セット → 全会一致チェック実行。

### admin（`x-admin-token` 配下。`/orgs/:eventId/actions/:actionId/whitelist/...`）

- `POST .../members/sync`
  → 現 role メンバー毎に `whitelist_members` 行（token 付き）を ensure。display_name は slack-names で解決。
    **削除メンバーの行は消さない**（または選択を明記）。
- `GET .../members`
  → display_name + 提出 status + token（リンクコピー用）の一覧。**リスト内容は絶対返さない。**
- `POST .../members/:memberId/rotate-token`
  → token 再生成。
- `GET .../results`
  → 既に通知済みの全会一致名前一覧。

## バックエンドサービス

`src/services/whitelist-consensus.ts` の
`checkConsensus(db, eventActionId, slackClient)`:

1. 設定 `roleId` の role メンバーをロード。
2. 未提出が 1 人でもいれば return。
3. 全員提出済みなら全 entries を復号 → 正規化 → 全メンバーで共通集合（intersection）を取る。
4. intersection の各名前で、まだ `whitelist_unanimous` に無いものは → INSERT + `notifyChannelId` に Slack 投稿
   （「全員が『<name>』を希望しています。誘いましょう。」— 個人特定は一切しない）。
5. fail-soft（Slack 失敗はログのみ）。

## フロントエンド

- **WhitelistFormPage**（メンバー向け、route `/whitelist/:token`）:
  GET でロード → フルネームを add/remove → POST で提出 → プライバシー注記
  「あなただけが閲覧・編集できます」を表示。
- **WhitelistAdminTab**（admin、`ActionMainContent` で action type `whitelist` のとき描画）:
  メンバー表（名前 / 提出状況 / copy-link / rotate）+ メンバー同期ボタン + 全会一致結果セクション。
  個人リストは非表示。
- **WhitelistSettingsForm**（admin、`ActionSettingsContent` 内）:
  workspace ドロップダウン + role ピッカー（どの role のメンバーか）+ 既存 `SingleChannelPicker` での通知チャンネル（workspaceId 必須）。
- **型登録**: フロント `EventActionType` union（`frontend/src/types/event.ts`）に `"whitelist"` 追加、
  `ACTION_META`（`frontend/src/lib/eventTabs.ts`、label「ホワイトリスト」、icon 🤝）追加。

## 設計セルフレビュー（3 人格）

### レビュアー #3: DB アーキテクト

| # | 指摘 | 対応・緩和 |
|---|------|-----------|
| 1 | `whitelist_entries` は名前 1 件 = 1 行のため、提出のたびに「全削除 → 全 INSERT」になる。同一 IV を使い回すと GCM の安全性が崩れるが、`encryptToken` は毎回ランダム IV を生成するので問題なし。トランザクション境界を明確化すること。 | POST は D1 batch（delete then insert）で原子的に置換。途中失敗時は submitted_at を更新しない順序にする。 |
| 2 | ON DELETE CASCADE の正しさ: action 削除で members/entries/unanimous が全消去。entries は members 経由の二段カスケード（action→members→entries）。SQLite で多段カスケードが効くには `PRAGMA foreign_keys=ON` が必要。 | D1/wrangler は外部キー制約が有効。既存テーブル（kejime 系）も同パターンで稼働実績あり。テストで「action 削除 → entries 消滅」を検証。 |
| 3 | インデックス: token は magic-link の主参照経路なので index（UNIQUE）必須。member_id は consensus の集計 JOIN で使うので index 必須。`(event_action_id, slack_user_id)` UNIQUE は sync の冪等性を担保。 | DDL に全て含める（PR1）。 |
| 4 | 復号集計コスト: consensus は「全メンバー × 全 entries」を復号。N×M 回の AES-GCM 復号は小規模（N≈数名）なら無視できるが、import key を毎回行うと無駄。 | `crypto.ts` は呼び出し毎に importKey するが、規模が小さいため許容。将来 N が増えたら key キャッシュを検討（YAGNI、今は不要）。 |
| 5 | unanimous 名前を**平文正規化で保存**する理由: その名前は Slack に公開ブロードキャスト済みで秘密ではない。平文にすることで `UNIQUE(event_action_id, name_normalized)` による dedup が DB レベルで成立し、競合状態（同時 POST 2 件）でも二重通知を物理的に防げる。 | INSERT 失敗（UNIQUE 違反）を「既通知」と解釈する idempotent 実装にする。 |
