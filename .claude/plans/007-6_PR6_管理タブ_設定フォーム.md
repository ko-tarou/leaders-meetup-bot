# PR6: 管理タブ + 設定フォーム

## 目的

admin 側のホワイトリスト管理 UI を実装する。
- メイン: メンバー表（名前 / 提出状況 / copy-link / rotate）+ メンバー同期 + 全会一致結果。
- 設定: workspace ドロップダウン + role ピッカー + 通知チャンネル（SingleChannelPicker）。
- `ActionMainContent` / `ActionSettingsContent` に `whitelist` の case を配線。
**個人リスト内容は一切表示しない。**

## 前提

- PR1（型 + ACTION_META）/ PR3（admin API）完了済み。理想は PR4（consensus）も完了
  （results 表示が意味を持つため）。
- 既存パターン: `ActionMainContent.tsx` / `ActionSettingsContent.tsx`（`frontend/src/pages/action-detail/`）が
  actionType ごとに switch し、未対応は default null を返す。
- 通知チャンネルは既存 `frontend/src/components/ui/SingleChannelPicker.tsx`（workspaceId 必須）。

## admin UI 原則（ID 禁止 / 内容非表示）

- 表示は全て **名前**（display_name）。token / slack_user_id / role id は UI に出さない。
- token はコピー操作のみ（クリップボード）。画面テキストには出さない。
- 個人リストの**内容・件数を一切表示しない**（PR3 の API がそもそも返さない）。
- 通知チャンネル / role は名前で選択・表示。

## 設計判断メモ

- **role ピッカーの対象**: 「宗教」イベントの `role_management` アクション配下に作られたロールから選ぶ。
  既存のロール一覧取得 API（roles ルータ）を流用する。
- **workspaceId と SingleChannelPicker の依存**: チャンネル候補は workspace の bot token 経由で取得するため、
  workspace 未選択時はチャンネルピッカーを無効化し、ヘルプテキストで誘導する。
- **rotate の確認**: 再発行すると旧リンクが失効するため、確認ダイアログを挟む（誤操作でメンバーのリンクを壊さない）。
