# 006-0 Phase 0: 安全網（詳細個別計画）

> 親計画: `006_大規模リファクタ_全体計画.md`
> このフェーズの大原則: **本番コード（`src/**`, `frontend/src/**`）を 1 行も変更しない**。
> 追加してよいのは devDependencies / テストファイル / テスト足場 / CI 設定 / `package.json` の script 行のみ。
> 既存データ・migration には一切触れない（テストは miniflare の使い捨て D1 に対して全 migration を流す）。

## 目的

構造変更（Phase 1 以降）に着手する前に、現状の振る舞いを "あるがまま" 固定する characterization テスト群と、それを動かすテスト基盤を整える。テスト 0 件・抽象化層なしの状態で大規模リファクタを行うと本番（Workers/D1）で回帰事故を起こすため、安全網を最優先で構築する。

## 「本番コード非変更」の担保方法（全 PR 共通）

各 PR の完了判定に以下を必須で含める:

1. `git diff --stat <base>..HEAD -- src frontend/src` の結果が **空**であること（本番コード差分ゼロ）
2. 追加されるのは `package.json`（script/devDeps 行のみ）/ `*.test.ts` / `test/` 配下 / `vitest.config.ts` / `.github/workflows/` のみ
3. characterization テストは「現状の出力」をアサートする（理想の仕様ではなく、今のコードが返す値をそのまま固定）
4. テストはアプリの公開関数を **そのまま import** して呼ぶ（テストのために本番コードへ export 追加が必要な場合のみ、その追加は最小限・振る舞い不変として #2/#6 レビューで個別承認）

> 注: 既存コードには既に純粋寄りの export が多数ある（例: `auto-cycle.ts` の `generateCandidateDates` / `generateCandidateDatesForFrequency` / `shouldStartPoll` / `shouldClosePoll`、`role-sync.ts` の `computeExpectedMembership` / `computeSyncDiff`、`role-auto-assign.ts` の `computeTargetRoleIds` / `expandWithAncestors`、`application-email.ts` の `readEmailTemplates` / `resolveTemplateIdForTrigger` / `renderSlackInviteLinks`、`participation-notification.ts` の `readParticipationNotificationsConfig`）。これらは export 追加なしでテスト可能。着手時に各サービスの export を再確認する。
